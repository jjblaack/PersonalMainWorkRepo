# 超级 Agent 概述与愿景

> 设计一个可 24 小时不间断运行、有记忆、能持续高质量完成目标的超级 Agent 系统。

---

## 1. 愿景

构建一个**对外表现为单一 Agent，对内包含管家、工人、任务层级结构**的超级 Agent 系统。这个系统能够：

- **24 小时不间断运行**：像真正的管家一样随时候命
- **持续记忆**：在"无记忆"的大模型 API 之上构建有状态的持续工作能力
- **任务闭环**：保证每个任务有始有终，质量可控
- **可扩展**：管家能够管理越来越多的工人和任务

---

## 2. 核心挑战

### 2.1 大模型的固有局限

大模型 API 本质上是**无状态的函数**：给定输入，产生输出，然后"遗忘"一切。

```
┌─────────────────────────────────────────────────────────┐
│  大模型 API                                              │
│  ┌─────────┐     ┌─────────┐                            │
│  │ Prompt  │ ──► │ Response│  (无状态，每次都是新的)    │
│  └─────────┘     └─────────┘                            │
└─────────────────────────────────────────────────────────┘
```

基于这样的"机器"构建持续工作的 Agent，需要解决：

1. **状态从哪里来？** → 外部持久化存储
2. **如何保持连续？** → 上下文管理和历史追踪
3. **如何保证质量？** → 监督机制和验收流程

### 2.2 24 小时运行的工程挑战

| 挑战 | 说明 |
|------|------|
| 进程保活 | 防止崩溃、自动重启 |
| 连接管理 | 断线重连、心跳保持 |
| 状态恢复 | 崩溃后从断点继续 |
| 资源管理 | 内存泄漏防护、Token 预算 |

### 2.3 任务闭环的管理挑战

| 挑战 | 说明 |
|------|------|
| 任务分解 | 如何将大任务拆分为可执行的子任务 |
| 进度追踪 | 如何知道每个任务的状态 |
| 依赖管理 | 如何处理任务间的先后关系 |
| 质量保证 | 如何验收任务完成 |
| 失败恢复 | 任务失败后如何处理 |

## 3. 从 Claude Code 学到的经验

### 3.1 架构分层

Claude Code 展现了一个清晰的分层架构：

```
┌─────────────────────────────────────────────────────────┐
│  入口层 (Entry Layer)                                    │
│  main.tsx, entrypoints/, bootstrap/                      │
└─────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────┐
│  命令层 (Command Layer)                                  │
│  commands.ts + commands/ 目录 (~88 个命令)                  │
└─────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────┐
│  工具层 (Tool Layer)                                     │
│  tools.ts + tools/ 目录 (~44 个工具) + Tool.ts 基类          │
└─────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────┐
│  服务层 (Service Layer)                                  │
│  services/ 目录 (API, MCP, OAuth, LSP, etc.)             │
└─────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────┐
│  基础设施层 (Infrastructure)                             │
│  types/, utils/, schemas/, migrations/                   │
└─────────────────────────────────────────────────────────┘
```

**关键启示**：分层清晰，每层职责明确，便于维护和扩展。

### 3.2 Bridge 系统 - 24 小时运行的参考

`src/bridge/` 目录展示了如何实现远程连接和持久化会话：

**核心机制**：
- **会话注册**：向服务器注册环境，获取 `environment_id` 和 `environment_secret`
- **轮询工作**：持续轮询服务器获取新任务
- **心跳保持**：定期发送心跳延长租约
- **断线重连**：连接失败时自动重连

```typescript
// 来自 src/bridge/types.ts

export type BridgeConfig = {
  dir: string
  machineName: string
  branch: string
  maxSessions: number
  sessionTimeoutMs?: number  // 会话超时
  // ...
}

export type BridgeApiClient = {
  registerBridgeEnvironment(config: BridgeConfig): Promise<{...}>
  pollForWork(environmentId: string, ...): Promise<WorkResponse | null>
  acknowledgeWork(...): Promise<void>
  heartbeatWork(...): Promise<{ lease_extended: boolean, state: string }>
  // ...
}
```

**关键启示**：
- 轮询模型比长连接更可靠
- 心跳机制防止会话过期
- 状态外置，崩溃后可恢复

### 3.3 记忆系统 - 持久化知识的参考

`src/memdir/memdir.ts` 展示了如何实现跨会话记忆：

**四种记忆类型**：
| 类型 | 用途 | 示例 |
|------|------|------|
| user | 用户角色、偏好 | "用户是数据科学家" |
| feedback | 用户反馈指导 | "测试不要用 mock" |
| project | 项目相关信息 | "认证模块重构中" |
| reference | 外部资源引用 | "Bug 追踪在 Linear" |

**记忆存储格式**：
```markdown
---
name: 记忆名称
description: 一行描述
type: user|feedback|project|reference
---

记忆内容

**Why:** 原因
**How to apply:** 如何应用
```

**关键启示**：
- 记忆分类型存储，便于检索
- 索引文件 (MEMORY.md) 统一入口
- 记忆文件独立，便于更新

### 3.4 任务系统 - 闭环管理的参考

`src/Task.ts` 展示了任务状态管理：

**任务状态机**：
```
pending ──► running ──► completed
    │           │          │
    │           │          │
    │           ▼          │
    └─────── failed/killed ◄──┘
```

**任务数据结构**：
```typescript
export type TaskStateBase = {
  id: string
  type: TaskType  // 'local_bash' | 'local_agent' | 'remote_agent' | ...
  status: TaskStatus  // 'pending' | 'running' | 'completed' | 'failed' | 'killed'
  description: string
  startTime: number
  endTime?: number
  outputFile: string
  outputOffset: number
  notified: boolean
}
```

**关键启示**：
- 任务状态显式定义
- 输出文件持久化
- 终端状态不可变

## 4. 超级 Agent 架构预览

基于 Claude Code 的经验，超级 Agent 的架构如下：

```
┌─────────────────────────────────────────────────────────┐
│                    超级 Agent (对外)                      │
└─────────────────────────────────────────────────────────┘
                              │
        ┌─────────────────────┼─────────────────────┐
        │                     │                     │
        ▼                     ▼                     ▼
┌───────────────┐     ┌───────────────┐     ┌───────────────┐
│    管家        │     │    管家        │     │    管家        │
│  (调度器)     │     │  (调度器)     │     │  (调度器)     │
│               │     │               │     │               │
│ ┌───────────┐ │     │ ┌───────────┐ │     │ ┌───────────┐ │
│ │ 指令接收  │ │     │ │ 指令接收  │ │     │ │ 指令接收  │ │
│ │ 任务分配  │ │     │ │ 任务分配  │ │     │ │ 任务分配  │ │
│ │ 进度监督  │ │     │ │ 进度监督  │ │     │ │ 进度监督  │ │
│ │ 结果验收  │ │     │ │ 结果验收  │ │     │ │ 结果验收  │ │
│ └───────────┘ │     │ └───────────┘ │     │ └───────────┘ │
└───────┬───────┘     └───────┬───────┘     └───────┬───────┘
        │                     │                     │
   ┌────┴────┐           ┌────┴────┐           ┌────┴────┐
   ▼    ▼    ▼           ▼    ▼    ▼           ▼    ▼    ▼
┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐
│工人│ │工人│ │工人│ │工人│ │工人│ │工人│ │工人│ │工人│ │工人│
└────┘ └────┘ └────┘ └────┘ └────┘ └────┘ └────┘ └────┘ └────┘
```

**核心组件**：
1. **管家 (Butler)**：调度器 + 监督器
2. **工人 (Worker)**：执行具体任务
3. **任务 (Task)**：工作单元
4. **记忆 (Memory)**：持久化知识
