# 关键技术决策清单

> 构建超级 Agent 过程中的关键技术决策点、选项分析和推荐方案。

## 1. 架构决策

### 1.1 单体 vs 微服务

**决策点**：超级 Agent 采用单体架构还是微服务架构？

| 选项 | 优点 | 缺点 | 推荐场景 |
|------|------|------|----------|
| **单体架构** | 简单、部署方便、调试容易 | 扩展性差、单点故障 | 个人使用、小规模部署 (推荐) |
| **微服务架构** | 独立扩展、故障隔离 | 复杂、运维成本高 | 大规模、多团队 |

**推荐**：单体架构

**理由**：
- 超级 Agent 的目标是简洁可靠
- 单机足以处理个人/小团队的工作负载
- 减少分布式系统的复杂性

**如何扩展**：
```
当需要扩展时，采用"单体 + 工人池"模式：
- 管家 (Butler) 保持单体
- 工人 (Worker) 可以分布式部署
```

### 1.2 长连接 vs 轮询

**决策点**：与服务器通信采用长连接还是轮询？

| 选项 | 优点 | 缺点 |
|------|------|------|
| **WebSocket 长连接** | 实时、双向 | 断线处理复杂、资源占用 |
| **HTTP 轮询** | 简单、可靠、易重试 | 延迟、额外请求 |
| **SSE (Server-Sent Events)** | 单向实时、简单 | 仅服务器→客户端 |

**推荐**：轮询模式

**理由**：
- Claude Code Bridge 系统采用轮询，证明其可靠性
- 断线后重试简单 (指数退避)
- 5 秒轮询间隔对大多数场景足够

**配置**：
```typescript
const POLL_CONFIG = {
  interval: 5000,         // 5 秒轮询间隔
  timeout: 30000,         // 30 秒超时
  backoff: [1, 2, 5, 10, 30],  // 退避序列 (秒)
}
```

### 1.3 同步 vs 异步任务执行

**决策点**：任务执行采用同步等待还是异步通知？

| 选项 | 优点 | 缺点 |
|------|------|------|
| **同步等待** | 简单、结果立即可用 | 阻塞、无法并行 |
| **异步通知** | 可并行、不阻塞 | 需要回调机制、复杂 |

**推荐**：异步通知

**理由**：
- 支持多任务并行
- 管家可以同时监督多个工人
- 符合 LLM 调用的实际时长

**实现模式**：
```typescript
// 任务提交
const taskId = await taskManager.createTask(spec)

// 订阅任务完成事件
eventBus.subscribe('task:completed', (event) => {
  if (event.taskId === taskId) {
    handleTaskComplete(event)
  }
})

// 同时可以做其他事情
await doOtherWork()
```

## 2. 存储决策

### 2.1 记忆存储格式

**决策点**：长期记忆采用什么格式存储？

| 选项 | 优点 | 缺点 |
|------|------|------|
| **Markdown 文件** | 可读、可版本控制、易编辑 | 检索效率低 |
| **SQLite 数据库** | 检索快、支持查询 | 需要数据库驱动 |
| **JSON 文件** | 易解析、结构化 | 不易人工阅读 |

**推荐**：Markdown 文件 (带 frontmatter)

**理由**：
- Claude Code 已验证可行
- 人类可读，便于调试
- 可用 git 版本控制

**格式**：
```markdown
---
name: 记忆名称
description: 一行描述
type: user|feedback|project|reference
createdAt: 2026-04-02
---

记忆内容

**Why:** 原因
**How to apply:** 如何应用
```

### 2.2 任务存储方式

**决策点**：任务状态和歷史如何存储？

| 选项 | 优点 | 缺点 |
|------|------|------|
| **JSONL 追加日志** | 写入快、可回放、易备份 | 查询需要加载全部 |
| **关系数据库** | 查询强大、支持索引 | 需要迁移管理 |
| **单个 JSON 文件** | 简单 | 并发写入冲突 |

**推荐**：JSONL 追加日志 + 定期快照

**理由**：
- 追加写入无锁冲突
- 可回放重建状态
- 快照提高查询效率

**实现**：
```typescript
// 追加日志
async function appendTaskEvent(event: TaskEvent): Promise<void> {
  const line = JSON.stringify({
    timestamp: Date.now(),
    ...event,
  })
  await appendFile('tasks.jsonl', line + '\n')
}

// 定期快照
async function createSnapshot(): Promise<void> {
  const state = await rebuildStateFromLog()
  await write('tasks-snapshot.json', JSON.stringify(state))
}

// 启动时从快照加载 + 重放增量
async function loadState(): Promise<TaskState> {
  const snapshot = await loadSnapshot()
  const events = await loadEventsSince(snapshot.timestamp)
  return applyEvents(snapshot, events)
}
```

## 3. LLM 集成决策

### 3.1 模型调用方式

**决策点**：如何调用 LLM API？

| 选项 | 优点 | 缺点 |
|------|------|------|
| **官方 SDK** | 稳定、功能全 | 依赖特定厂商 |
| **直接 HTTP 调用** | 灵活、可切换供应商 | 需要自己处理重试等 |
| **统一接口层** | 可切换、易测试 | 增加抽象层 |

**推荐**：统一接口层

**理由**：
- 避免供应商锁定
- 便于切换不同模型
- 便于测试 (可注入 mock)

**实现**：
```typescript
// 定义统一接口
interface LLMProvider {
  chat(messages: Message[], options?: ChatOptions): Promise<ChatResponse>
  countTokens(text: string): number
}

// Anthropic 实现
class AnthropicProvider implements LLMProvider {
  async chat(messages, options) {
    const client = new Anthropic(API_KEY)
    return client.messages.create({ ... })
  }
}

// OpenAI 实现 (备用)
class OpenAIProvider implements LLMProvider {
  async chat(messages, options) {
    const client = new OpenAI(API_KEY)
    return client.chat.completions.create({ ... })
  }
}

// 工厂函数
function createProvider(type: 'anthropic' | 'openai'): LLMProvider {
  switch (type) {
    case 'anthropic': return new AnthropicProvider()
    case 'openai': return new OpenAIProvider()
  }
}
```

### 3.2 上下文窗口管理

**决策点**：如何管理有限的上下文窗口？

| 选项 | 优点 | 缺点 |
|------|------|------|
| **截断最早的消息** | 简单 | 可能丢失重要信息 |
| **智能摘要** | 保留更多信息 | 需要额外 API 调用 |
| **分层上下文** | 重要信息优先 | 实现复杂 |

**推荐**：分层上下文 + 按需摘要

**理由**：
- 系统提示和记忆永远保留
- 最近对话保持完整
- 早期对话可摘要

**实现**：
```typescript
async function buildContext(
  systemPrompt: string,
  memories: Memory[],
  conversation: Message[],
  maxTokens: number
): Promise<Message[]> {
  const messages: Message[] = [
    { role: 'system', content: systemPrompt },
  ]
  
  // 注入记忆
  if (memories.length > 0) {
    messages.push({
      role: 'system',
      content: buildMemoryContext(memories),
    })
  }
  
  // 添加对话 (从最近开始)
  let tokenCount = countTokens(messages)
  
  for (const msg of conversation.reverse()) {
    const msgTokens = countTokens(msg)
    if (tokenCount + msgTokens > maxTokens * 0.9) {
      // 接近限制，摘要最早的消息
      const summary = await summarize(messages.slice(2, 10))
      messages.splice(2, 8, { role: 'user', content: summary })
      break
    }
    messages.push(msg)
    tokenCount += msgTokens
  }
  
  return messages
}
```

### 3.3 Token 预算管理

**决策点**：如何控制 API 调用成本？

| 选项 | 优点 | 缺点 |
|------|------|------|
| **无限制** | 简单 | 成本不可控 |
| **日预算** | 成本可控 | 可能提前用完 |
| **任务预算** | 精细控制 | 实现复杂 |

**推荐**：日预算 + 任务估算

**理由**：
- 日预算防止意外高额账单
- 任务估算帮助优先级决策

**实现**：
```typescript
class TokenBudget {
  private dailyLimit: number
  private usedToday: number = 0
  private taskEstimates: Map<string, number> = new Map()
  
  constructor(dailyLimit: number) {
    this.dailyLimit = dailyLimit
  }
  
  // 检查任务是否可以执行
  canExecuteTask(taskId: string, estimatedTokens: number): boolean {
    const remaining = this.getRemaining()
    return estimatedTokens <= remaining
  }
  
  // 使用 Token
  use(tokens: number, taskId?: string): void {
    if (!this.canUse(tokens)) {
      throw new BudgetExceededError(
        `Token budget exceeded: ${tokens} > ${this.getRemaining()}`
      )
    }
    this.usedToday += tokens
    
    if (taskId) {
      const prev = this.taskEstimates.get(taskId) || 0
      this.taskEstimates.set(taskId, prev + tokens)
    }
  }
  
  getRemaining(): number {
    return this.dailyLimit - this.usedToday
  }
  
  // 获取任务实际使用
  getTaskUsage(taskId: string): number {
    return this.taskEstimates.get(taskId) || 0
  }
}
```

## 4. 并发决策

### 4.1 最大并发工人数

**决策点**：如何决定同时运行的工人数量？

| 选项 | 优点 | 缺点 |
|------|------|------|
| **固定数量** | 简单可预测 | 不能适应负载 |
| **动态调整** | 适应负载 | 实现复杂 |
| **基于资源** | 合理 | 需要监控 |

**推荐**：基于资源的动态调整

**公式**：
```typescript
function calculateMaxWorkers(): number {
  const cpuCores = os.cpus().length
  const memoryGB = os.totalmem() / (1024 * 1024 * 1024)
  
  // 每个工人约需 200MB 内存
  const memoryLimit = Math.floor(memoryGB / 0.2)
  
  // CPU 限制：每个核心 2 个工人
  const cpuLimit = cpuCores * 2
  
  // 取较小值，但不超过上限
  return Math.min(cpuLimit, memoryLimit, 10)
}
```

### 4.2 任务优先级调度

**决策点**：如何调度有优先级的任务？

| 选项 | 优点 | 缺点 |
|------|------|------|
| **FIFO** | 简单公平 | 紧急任务需等待 |
| **优先级队列** | 紧急任务优先 | 低优先级可能饿死 |
| **时间片轮转** | 公平 | 实现复杂 |

**推荐**：优先级队列 +  starvation 保护

**实现**：
```typescript
class PriorityTaskQueue {
  private queues: Map<Priority, Task[]> = new Map([
    ['urgent', []],
    ['high', []],
    ['normal', []],
    ['low', []],
  ])
  
  private waitTimes: Map<Priority, number> = new Map()
  
  enqueue(task: Task): void {
    const queue = this.queues.get(task.priority)!
    queue.push(task)
  }
  
  dequeue(): Task | null {
    // 更新等待时间
    for (const [priority, queue] of this.queues.entries()) {
      if (queue.length > 0) {
        this.waitTimes.set(priority, (this.waitTimes.get(priority) || 0) + 1)
      }
    }
    
    // 检查是否有任务等待太久 (starvation 保护)
    for (const [priority, time] of this.waitTimes.entries()) {
      if (time > STARVATION_THRESHOLD) {
        const queue = this.queues.get(priority)!
        if (queue.length > 0) {
          this.waitTimes.set(priority, 0)
          return queue.shift()!
        }
      }
    }
    
    // 按优先级获取
    for (const priority of ['urgent', 'high', 'normal', 'low']) {
      const queue = this.queues.get(priority)!
      if (queue.length > 0) {
        return queue.shift()!
      }
    }
    
    return null
  }
}
```

## 5. 容错决策

### 5.1 重试策略

**决策点**：失败后如何重试？

| 选项 | 优点 | 缺点 |
|------|------|------|
| **立即重试** | 快速恢复 | 可能雪崩 |
| **固定间隔** | 简单 | 不够灵活 |
| **指数退避** | 避免雪崩、适应不同情况 | 实现稍复杂 |

**推荐**：指数退避 + 抖动

**实现**：
```typescript
async function retryWithBackoff<T>(
  fn: () => Promise<T>,
  options?: RetryOptions
): Promise<T> {
  const maxRetries = options?.maxRetries ?? 3
  const baseDelay = options?.baseDelay ?? 1000
  const maxDelay = options?.maxDelay ?? 30000
  
  let lastError: Error
  
  for (let i = 0; i < maxRetries; i++) {
    try {
      return await fn()
    } catch (error) {
      lastError = error as Error
      
      // 检查是否可重试
      if (!isRetryable(error)) {
        throw error
      }
      
      // 计算延迟 (指数退避 + 抖动)
      const delay = Math.min(
        baseDelay * Math.pow(2, i),
        maxDelay
      )
      
      // 添加 0-20% 的抖动
      const jitter = delay * 0.2 * Math.random()
      
      logger.info(
        `Retry ${i + 1}/${maxRetries} after ${delay + jitter}ms`
      )
      
      await sleep(delay + jitter)
    }
  }
  
  throw lastError!
}
```

### 5.2 优雅降级

**决策点**：资源不足时如何降级？

| 选项 | 优点 | 缺点 |
|------|------|------|
| **直接失败** | 简单 | 用户体验差 |
| **排队等待** | 最终能执行 | 响应慢 |
| **分级降级** | 平衡体验和成本 | 实现复杂 |

**推荐**：分级降级

**降级策略**：
```typescript
class GracefulDegradation {
  async execute(task: Task): Promise<void> {
    try {
      await this.executeWithFullCapability(task)
    } catch (error) {
      if (error instanceof TokenBudgetExceeded) {
        // 降级：使用更便宜的模型
        await this.executeWithCheaperModel(task)
      } else if (error instanceof WorkerExhausted) {
        // 降级：排队等待
        await this.queueTask(task)
      } else if (error instanceof MemoryLimit) {
        // 降级：清理缓存后重试
        await this.cleanupAndRetry(task)
      } else {
        throw error
      }
    }
  }
}
```

## 6. 安全决策

### 6.1 工具权限控制

**决策点**：如何控制 Agent 可以使用的工具？

| 选项 | 优点 | 缺点 |
|------|------|------|
| **全权限** | 简单 | 风险高 |
| **白名单** | 安全 | 需要预先配置 |
| **按需审批** | 灵活且安全 | 需要人工介入 |

**推荐**：白名单 + 高危操作审批

**实现**：
```typescript
// 工具权限级别
enum ToolPermissionLevel {
  LOW = 'low',      // 读取操作，自动批准
  MEDIUM = 'medium', // 写入操作，需要确认
  HIGH = 'high',    // 删除/执行，需要明确批准
}

// 工具权限定义
const TOOL_PERMISSIONS: Record<string, ToolPermissionLevel> = {
  'FileRead': ToolPermissionLevel.LOW,
  'FileWrite': ToolPermissionLevel.MEDIUM,
  'Bash': ToolPermissionLevel.HIGH,
  'FileDelete': ToolPermissionLevel.HIGH,
}

// 权限检查
async function checkToolPermission(
  toolName: string,
  input: unknown
): Promise<PermissionResult> {
  const level = TOOL_PERMISSIONS[toolName]
  
  if (level === ToolPermissionLevel.LOW) {
    return { approved: true }
  }
  
  if (level === ToolPermissionLevel.MEDIUM) {
    // 检查是否在自动批准列表
    if (isInAutoApproveList(toolName, input)) {
      return { approved: true }
    }
    return { approved: false, reason: 'Requires confirmation' }
  }
  
  // HIGH 级别总是需要明确批准
  return { approved: false, reason: 'Requires explicit approval' }
}
```
