# 核心挑战与解决思路

> 深入分析构建 24 小时超级 Agent 面临的核心挑战，以及从 Claude Code 项目中汲取的解决方案。

---

## 1. 挑战一：如何在无状态的大模型 API 之上构建有状态的 Agent？

### 1.1 问题本质

大模型 API 是一个纯函数：
```
Response = f(Prompt)
```

每次调用都是独立的，模型没有"上一次对话"的概念。

### 1.2 Claude Code 的解决方案

**外部状态存储**：将所有状态存储在 API 之外。

```typescript
// src/QueryEngine.ts - 核心状态管理
export class QueryEngine {
  private mutableMessages: Message[]    // 对话历史
  private readFileState: FileStateCache // 文件状态缓存
  private totalUsage: NonNullableUsage  // Token 使用追踪
  private permissionDenials: SDKPermissionDenial[] // 权限拒绝记录
  
  // 每轮对话 persist
  async submitMessage(newMessage: Message) {
    this.mutableMessages.push(newMessage)
    await flushSessionStorage()  // 持久化到磁盘
  }
}
```

**会话持久化**：
```typescript
// src/utils/sessionStorage.ts
export function flushSessionStorage() {
  // 将内存状态写入 .jsonl 文件
  const transcriptPath = join(getProjectDir(), 'transcript.jsonl')
  writeFileSync(transcriptPath, JSON.stringify(messages))
}
```

### 1.3 对超级 Agent 的启示

**状态外置三原则**：
1. **关键状态必须落盘**：崩溃后可恢复
2. **状态变更即记录**：每条变更追加到日志
3. **状态查询可重建**：从日志回放重建状态

**超级 Agent 的状态设计**：
```
┌─────────────────────────────────────────────────────────┐
│                    超级 Agent 状态                        │
├─────────────────────────────────────────────────────────┤
│  短期状态 (内存)                                         │
│  - 当前正在执行的任务                                    │
│  - 工人的实时状态                                        │
│  - 临时上下文                                            │
├─────────────────────────────────────────────────────────┤
│  长期状态 (磁盘)                                         │
│  - 任务历史 (.jsonl 追加日志)                             │
│  - 记忆文件 (MEMORY.md + memory/*.md)                   │
│  - 配置和进度                                            │
└─────────────────────────────────────────────────────────┘
```

## 2. 挑战二：如何保证 24 小时不间断运行？

### 2.1 问题本质

- 进程可能崩溃
- 网络连接可能断开
- 资源可能耗尽

### 2.2 Claude Code Bridge 系统的解决方案

**轮询 + 心跳模型**：
```typescript
// src/bridge/types.ts
export type BridgeApiClient = {
  // 注册环境
  registerBridgeEnvironment(config: BridgeConfig): Promise<{
    environment_id: string
    environment_secret: string
  }>
  
  // 轮询工作
  pollForWork(
    environmentId: string,
    signal?: AbortSignal,
  ): Promise<WorkResponse | null>
  
  // 心跳保活
  heartbeatWork(
    environmentId: string,
    workId: string,
    sessionToken: string,
  ): Promise<{ lease_extended: boolean, state: string }>
}

// 会话超时配置
export const DEFAULT_SESSION_TIMEOUT_MS = 24 * 60 * 60 * 1000  // 24 小时
```

**断线重连逻辑**：
```typescript
// 轮询循环 (推断)
async function pollLoop() {
  while (true) {
    try {
      const work = await api.pollForWork(envId, abortSignal)
      if (work) {
        await handleWork(work)
      }
    } catch (error) {
      // 网络错误，等待后重试
      await sleep(calculateBackoff())
      continue
    }
    
    // 定期发送心跳
    await api.heartbeatWork(envId, currentWorkId, token)
  }
}
```

### 2.3 对超级 Agent 的启示

**24 小时运行架构**：
```
┌─────────────────────────────────────────────────────────┐
│                    守护进程                              │
├─────────────────────────────────────────────────────────┤
│  主循环                                                  │
│  ┌─────────────────────────────────────────────────┐    │
│  │ 1. 检查连接状态                                   │    │
│  │ 2. 轮询新指令/消息                               │    │
│  │ 3. 处理指令                                      │    │
│  │ 4. 发送心跳 (如连接正常)                         │    │
│  │ 5. 检查资源使用 (内存、Token)                    │    │
│  │ 6. 回到步骤 1                                     │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  异常处理                                                │
│  - 网络错误：指数退避重连                               │
│  - 进程崩溃：自动重启，从最后状态恢复                   │
│  - 资源耗尽：清理 + 告警                                 │
└─────────────────────────────────────────────────────────┘
```

**关键设计决策**：
| 设计点 | 选项 | 推荐 | 理由 |
|--------|------|------|------|
| 连接方式 | 长连接 vs 轮询 | 轮询 | 更可靠，断线影响小 |
| 心跳间隔 | 30s / 60s / 120s | 60s | 平衡实时性和开销 |
| 会话超时 | 1h / 24h / 无限 | 24h | 符合"天"的周期 |
| 重连策略 | 立即 / 固定间隔 / 指数退避 | 指数退避 | 避免雪崩 |

## 3. 挑战三：如何保证任务有始有终？

### 3.1 问题本质

- 任务可能被遗忘
- 进度可能丢失
- 失败后无恢复

### 3.2 Claude Code 任务系统的解决方案

**任务状态机**：
```typescript
// src/Task.ts
export type TaskStatus =
  | 'pending'     // 等待执行
  | 'running'     // 执行中
  | 'completed'   // 已完成
  | 'failed'      // 失败
  | 'killed'      // 被停止

export function isTerminalTaskStatus(status: TaskStatus): boolean {
  return status === 'completed' || status === 'failed' || status === 'killed'
}

// 任务基础结构
export type TaskStateBase = {
  id: string
  type: TaskType
  status: TaskStatus
  description: string
  startTime: number
  endTime?: number
  outputFile: string      // 输出文件路径 (持久化)
  outputOffset: number    // 输出读取偏移
  notified: boolean       // 是否已通知
}
```

**任务通知机制**：
```typescript
// <task-notification> XML 格式
// src/coordinator/coordinatorMode.ts

<task-notification>
  <task-id>{agentId}</task-id>
  <status>completed|failed|killed</status>
  <summary>{人类可读的状态摘要}</summary>
  <result>{代理的最终文本响应}</result>
  <usage>
    <total_tokens>N</total_tokens>
    <tool_uses>N</tool_uses>
    <duration_ms>N</duration_ms>
  </usage>
</task-notification>
```

### 3.3 对超级 Agent 的启示

**任务闭环设计**：
```
任务生命周期：
pending ──► running ──► completed
   │           │           │
   │           │           │
   │           ▼           │
   │       failed ─────────┤
   │           │           │
   │           ▼           │
   └─────── killed ◄───────┘

关键设计：
1. 每个任务必须有终端状态 (completed/failed/killed)
2. 终端状态不可逆 (防止状态混乱)
3. 任务输出必须持久化 (outputFile)
4. 状态变更必须通知 (notified 标志)
```

**超级 Agent 的任务管理**：
```typescript
// 伪代码示例
class TaskManager {
  private tasks: Map<string, Task>
  
  async createTask(spec: TaskSpec): Promise<string> {
    const task = {
      id: generateId(),
      subject: spec.subject,
      description: spec.description,
      status: 'pending',
      createdAt: Date.now(),
      outputFile: this.getOutputPath(taskId),
    }
    this.tasks.set(task.id, task)
    await this.persistTask(task)  // 立即持久化
    return task.id
  }
  
  async updateTaskStatus(taskId: string, status: TaskStatus) {
    const task = this.tasks.get(taskId)
    if (isTerminalTaskStatus(task.status)) {
      throw new Error('Cannot modify terminal task')
    }
    task.status = status
    task.updatedAt = Date.now()
    if (isTerminalTaskStatus(status)) {
      task.endTime = Date.now()
    }
    await this.persistTask(task)
    this.notifyStatusChange(task)
  }
}
```

## 4. 挑战四：如何实现有效的记忆系统？

### 4.1 问题本质

- 大模型没有长期记忆
- 上下文窗口有限
- 如何区分"需要记忆"和"可以遗忘"的信息

### 4.2 Claude Code 记忆系统的解决方案

**四种记忆类型**：
```typescript
// src/memdir/memdir.ts

// 记忆类型定义
type MemoryType = 'user' | 'feedback' | 'project' | 'reference'

// 记忆文件格式
---
name: 记忆名称
description: 一行描述
type: user|feedback|project|reference
---

记忆内容

**Why:** 原因
**How to apply:** 如何应用
```

**记忆提取规则**：
```typescript
// 什么应该保存
function shouldSave(memory: PotentialMemory): boolean {
  // 应该保存
  if (memory.type === 'user' && isNonObvious) return true
  if (memory.type === 'feedback') return true  // 用户反馈通常有价值
  if (memory.type === 'project' && hasLongTermValue) return true
  if (memory.type === 'reference') return true
  
  // 不应该保存
  if (memory.derivableFromCode) return false  // 可从代码推导
  if (memory.inGitHistory) return false       // Git 历史已有
  if (memory.type === 'ephemeral') return false  // 临时状态
  
  return false
}
```

**索引文件管理**：
```typescript
// MEMORY.md 格式
- [Title](file.md) — one-line hook
- [用户是数据科学家](user_data_scientist.md) — 用户背景
- [测试不要用 mock](feedback_no_mock.md) — 测试偏好
```

### 4.3 对超级 Agent 的启示

**分层记忆模型**：
```
┌─────────────────────────────────────────────────────────┐
│                    记忆分层模型                          │
├─────────────────────────────────────────────────────────┤
│  工作记忆 (Working Memory)                               │
│  - 当前任务的上下文                                      │
│  - 短期暂存区                                            │
│  - 容量有限，任务完成后清空                             │
├─────────────────────────────────────────────────────────┤
│  短期记忆 (Short-term Memory)                            │
│  - 最近 N 次对话的历史                                    │
│  - 滑动窗口维护                                          │
│  - 超出窗口后压缩或丢弃                                 │
├─────────────────────────────────────────────────────────┤
│  长期记忆 (Long-term Memory)                             │
│  - 用户偏好、反馈                                        │
│  - 项目背景、决策                                        │
│  - 外部资源引用                                          │
│  - 持久化存储，按需检索                                 │
└─────────────────────────────────────────────────────────┘
```

**记忆检索策略**：
1. **启动时**：加载所有长期记忆到上下文
2. **任务创建时**：检索相关记忆注入任务上下文
3. **任务执行中**：可主动查询记忆系统

## 5. 挑战五：如何管理越来越多的工人和任务？

- 工人数量增加，管理复杂度上升

- 资源竞争和冲突

### 5.2 Claude Code 协调器模式的解决方案

**协调器职责**：

## 6. 总结：核心设计原则

| 原则 | 说明 |
|------|------|
| **状态外置** | 所有关键状态存储在外部，崩溃后可恢复 |
| **轮询 + 心跳** | 可靠的连接管理，断线自动重连 |
| **任务闭环** | 每个任务必须有始有终，状态可追踪 |
| **记忆分层** | 工作记忆、短期记忆、长期记忆分离 |
| **监督分离** | 管家不干活，专注于调度和监督 |
| **简洁优先** | 越简单的系统越可靠 |
