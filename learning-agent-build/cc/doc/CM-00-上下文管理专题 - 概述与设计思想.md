# 上下文管理专题 - 概述与设计思想

## 1. 引言

### 1.1 什么是上下文管理

在 LLM 驱动的代码助手系统中，**上下文管理**是指：在有限的 token 预算和上下文窗口内，智能地组织、压缩、持久化和恢复与用户交互相关的所有信息，以确保模型始终拥有完成任务所需的必要上下文，同时避免上下文膨胀导致的性能下降和成本增加。

### 1.2 为什么需要上下文管理

LLM 的上下文窗口是有限的（例如 Claude 模型的上下文窗口限制）。随着对话的进行，如果不加管理，上下文会不断增长直到超出限制。上下文管理的核心挑战：

1. **Token 预算约束**：每次 API 调用都有 token 成本，需要在有限预算内最大化有用信息
2. **上下文窗口限制**：模型能处理的输入长度有上限
3. **信息相关性衰减**：早期对话内容对当前任务的相关性通常较低
4. **跨会话连续性**：用户可能中断对话后恢复，需要保留关键状态
5. **多项目/多模式切换**：用户可能在不同项目、不同工作模式间切换

### 1.3 Claude Code 的上下文管理哲学

Claude Code 的上下文管理遵循以下核心原则：

| 原则 | 描述 |
|------|------|
| **分层注入** | 从系统级到用户级到项目级，多层上下文按优先级注入 |
| **有损压缩** | 保留语义摘要，丢弃细节，通过恢复机制按需取回细节 |
| **惰性加载** | 只在需要时加载上下文，避免预加载浪费 |
| **状态外置** | 将大状态存储在磁盘/内存缓存中，上下文中只存引用 |
| **边界标记** | 在压缩点设置清晰的边界，便于追溯和恢复 |

## 3. 核心组件概览

### 3.1 CLAUDE.md 上下文注入系统

**文件位置**: `src/utils/claudemd.ts` (1480 行)

**核心职责**:
- 发现并加载多层级 CLAUDE.md 文件
- 解析 @include 指令，支持循环引用检测
- 处理条件规则（frontmatter 中的 paths glob 匹配）
- 合并文件内容，按优先级排序

**文件层级（从高到低）**:
1. **Managed**: `/etc/claude-code/CLAUDE.md` - 系统级配置
2. **User**: `~/.claude/CLAUDE.md` - 用户级配置
3. **Project**: `CLAUDE.md`, `.claude/CLAUDE.md`, `.claude/rules/*.md` - 项目级配置
4. **Local**: `CLAUDE.local.md` - 本地临时配置

**关键机制**:
- `@include` 指令支持相对路径、绝对路径、家目录路径
- 最大包含深度：5 层（防止无限递归）
- 条件规则：基于 glob 匹配当前文件路径决定是否应用规则

### 3.2 会话压缩系统

**文件位置**: `src/services/compact/compact.ts`

**核心职责**:
- 执行 PreCompact hooks
- 调用 forked agent 生成摘要（复用 prompt cache）
- 生成 post-compact attachments（文件、skill、plan）
- 创建 boundary marker 和 summary messages
- 执行 SessionStart 和 PostCompact hooks
- 记录 telemetry

**压缩类型**:
| 类型 | 触发条件 | 行为 |
|------|----------|------|
| **Auto Compact** | 上下文接近 token 限制 | 自动压缩最早的消息 |
| **Manual Compact** | 用户显式请求 | 压缩指定范围的消息 |
| **Full Compact** | 完整对话压缩 | 生成完整摘要，清空原消息 |
| **Partial Compact** | 保留部分消息 | 'from' 保留 prefix，'up_to' 保留 suffix |

**Token 预算（压缩后）**:
```typescript
POST_COMPACT_MAX_FILES_TO_RESTORE = 5
POST_COMPACT_TOKEN_BUDGET = 50_000
POST_COMPACT_MAX_TOKENS_PER_FILE = 5_000
POST_COMPACT_MAX_TOKENS_PER_SKILL = 5_000
POST_COMPACT_SKILLS_TOKEN_BUDGET = 25_000
```

### 3.3 历史记录系统

**文件位置**: `src/history.ts`

**核心职责**:
- 记录用户命令历史（JSONL 格式）
- 支持跨会话历史共享
- 大内容外部存储（hash-based paste store）
- 双缓冲机制（pendingEntries + 磁盘存储）

**存储格式**:
```typescript
type LogEntry = {
  display: string
  pastedContents: Record<number, StoredPastedContent>
  timestamp: number
  project: string
  sessionId?: string
}
```

### 3.4 会话持久化系统

**文件位置**: `src/utils/sessionStorage.ts`

**核心职责**:
- JSONL 格式会话存储
- 会话元数据持久化（custom title、tag、agent setting）
- 会话恢复和续期
- Bridge 远程会话同步

## 5. 关键设计决策

### 5.1 为什么使用 JSONL 格式存储会话？

**优势**:
- **增量写入**：每条消息独立成行，支持流式追加
- **容错性**：单行损坏不影响其他消息
- **易解析**：每行是完整的 JSON 对象，便于调试
- **tail 友好**：可以用 `tail -n 100` 快速查看最近消息

### 5.2 为什么压缩后要恢复文件？

压缩会丢弃具体的文件内容，只保留摘要。但用户刚刚读取的文件很可能在后续对话中继续用到。恢复策略：

1. 从 `readFileState` 缓存中获取最近读取的文件
2. 跳过 preservedMessages 中已有的 Read 结果（避免重复）
3. 按 token 预算限制恢复的文件数量（默认最多 5 个）

### 5.3 为什么粘贴内容要外部存储？

大段粘贴内容（如文件内容、长文本）如果直接存储在历史记录中，会导致：
- 历史记录文件迅速膨胀
- 跨会话同步效率低下

**解决方案**:
- 内容超过阈值 → 计算 hash → 外部存储
- 历史记录中只存储 hash 引用
- 需要时通过 hash 反查内容

### 5.4 为什么需要多层级 CLAUDE.md？

不同层级的配置有不同的作用域和优先级：

| 层级 | 作用域 | 典型用途 |
|------|--------|----------|
| Managed | 全系统 | IT 部门强制策略、安全规范 |
| User | 单用户 | 个人编码风格偏好、常用工具配置 |
| Project | 单项目 | 项目特定的构建命令、代码规范 |
| Local | 临时任务 | 当前任务的临时笔记、待办事项 |

## 7. 总结

Claude Code 的上下文管理系统是一个多层次、多策略的综合系统：

1. **静态上下文**通过 CLAUDE.md 多层级注入系统管理
2. **动态上下文**通过会话压缩机制控制增长
3. **大状态外置**通过文件缓存、历史存储、哈希粘贴实现
4. **状态可恢复**通过边界标记、缓存引用、元数据持久化保证

这套系统的设计核心是**在有限 token 预算内最大化有用上下文**，通过分层、压缩、外置、恢复四大策略实现这一目标。
