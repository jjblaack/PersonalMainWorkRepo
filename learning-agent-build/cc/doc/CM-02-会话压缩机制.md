# CM-02 - 会话压缩机制

## 1. 概述

会话压缩（Compaction）是 Claude Code 管理上下文窗口的核心机制。当对话进行到一定程度，上下文 token 数量接近模型限制时，系统会自动或手动触发压缩，将早期对话内容压缩成摘要，释放上下文空间。

### 1.1 核心目标

1. **控制上下文增长**：防止 token 数量超出模型限制
2. **保留关键信息**：通过摘要保留早期对话的语义
3. **恢复必要状态**：压缩后恢复最近读取的文件等状态
4. **最小化信息损失**：通过多种策略减少压缩带来的上下文丢失

### 1.2 压缩类型

| 类型 | 触发条件 | 行为 |
|------|----------|------|
| **Auto Compact** | 上下文接近 token 限制（默认阈值） | 自动压缩最早的消息 |
| **Manual Compact** | 用户显式请求（如 `/compact` 命令） | 压缩指定范围的消息 |
| **Full Compact** | 完整对话压缩 | 生成完整摘要，清空原消息 |
| **Partial Compact** | 用户选择某条消息作为pivot | 'from' 保留 prefix，'up_to' 保留 suffix |

## 2. Token 预算管理

### 2.1 压缩后恢复的 Token 预算

```typescript
// 核心常量定义
export const POST_COMPACT_MAX_FILES_TO_RESTORE = 5      // 最多恢复 5 个文件
export const POST_COMPACT_TOKEN_BUDGET = 50_000         // 文件恢复总预算
export const POST_COMPACT_MAX_TOKENS_PER_FILE = 5_000   // 单文件预算
export const POST_COMPACT_MAX_TOKENS_PER_SKILL = 5_000  // 单 skill 预算
export const POST_COMPACT_SKILLS_TOKEN_BUDGET = 25_000  // Skills 总预算
```

**设计 rationale**：

- **文件恢复预算 50K**：足够恢复 5 个中等大小的文件（每个约 10K tokens）
- **单文件 5K**：防止单个大文件占用过多预算
- **Skills 预算 25K**：约可容纳 5 个 skills（每个 5K），per-skill 截断优于整体丢弃

### 2.2 压缩触发阈值

自动压缩的触发由 `autoCompactThreshold` 控制：

```typescript
// 当上下文 token 数量超过阈值时触发
if (currentTokenCount >= autoCompactThreshold) {
  await autoCompactIfNeeded()
}
```

**压缩后评估**：
```typescript
const willRetriggerNextTurn =
  recompactionInfo !== undefined &&
  truePostCompactTokenCount >= recompactionInfo.autoCompactThreshold
```

如果压缩后 token 数量仍然超过阈值，下一轮会再次触发压缩。

## 4. 部分压缩（Partial Compact）

### 4.1 两种方向

```typescript
type PartialCompactDirection = 'from' | 'up_to'

// 'from': 压缩 pivot 之后的消息，保留之前的
//   [kept messages] [summary of later messages]
//   Prompt cache 保留（kept messages 在前）

// 'up_to': 压缩 pivot 之前的消息，保留之后的
//   [summary of earlier messages] [kept messages]
//   Prompt cache 失效（summary 在 kept 之前）
```

### 4.2 消息分割

```typescript
const messagesToSummarize =
  direction === 'up_to'
    ? allMessages.slice(0, pivotIndex)
    : allMessages.slice(pivotIndex)

const messagesToKeep =
  direction === 'up_to'
    ? allMessages
        .slice(pivotIndex)
        .filter(m =>
          m.type !== 'progress' &&
          !isCompactBoundaryMessage(m) &&
          !(m.type === 'user' && m.isCompactSummary),
        )
    : allMessages.slice(0, pivotIndex).filter(m => m.type !== 'progress')
```

**注意**：
- 'up_to' 必须过滤掉旧的 compact boundaries/summaries
- 'from' 保留旧的 summary（它们的 covered history 仍然需要）

### 4.3 Boundary 注解

部分压缩需要记录 preserved segment 信息，用于消息链的 relink：

```typescript
export function annotateBoundaryWithPreservedSegment(
  boundary: SystemCompactBoundaryMessage,
  anchorUuid: UUID,
  messagesToKeep: readonly Message[] | undefined,
): SystemCompactBoundaryMessage {
  const keep = messagesToKeep ?? []
  if (keep.length === 0) return boundary
  
  return {
    ...boundary,
    compactMetadata: {
      ...boundary.compactMetadata,
      preservedSegment: {
        headUuid: keep[0]!.uuid,
        anchorUuid,  // 'from': boundary; 'up_to': last summary
        tailUuid: keep.at(-1)!.uuid,
      },
    },
  }
}
```

**Purpose**：loader 使用这些信息正确重建消息链的 parentUuids 关系。

## 5. 压缩后文件恢复

### 5.1 问题背景

压缩会丢弃具体的文件内容，只保留摘要。但用户刚刚读取的文件很可能在后续对话中继续用到。如果不恢复，模型会"忘记"这些文件的内容。

### 5.2 恢复策略

```typescript
export async function createPostCompactFileAttachments(
  readFileState: Record<string, { content: string; timestamp: number }>,
  toolUseContext: ToolUseContext,
  maxFiles: number = POST_COMPACT_MAX_FILES_TO_RESTORE,
  preservedMessages: Message[] = [],
): Promise<AttachmentMessage[]> {
  // 1. 收集 preservedMessages 中已有的 Read 结果路径（避免重复）
  const preservedReadPaths = collectReadToolFilePaths(preservedMessages)
  
  // 2. 按时间排序，选择最近的文件
  const recentFiles = Object.entries(readFileState)
    .map(([filename, state]) => ({ filename, ...state }))
    .filter(file =>
      !shouldExcludeFromPostCompactRestore(file.filename, toolUseContext.agentId) &&
      !preservedReadPaths.has(expandPath(file.filename)),  // 跳过已有的
    )
    .sort((a, b) => b.timestamp - a.timestamp)
    .slice(0, maxFiles)
  
  // 3. 并行读取文件
  const results = await Promise.all(
    recentFiles.map(async file => {
      const attachment = await generateFileAttachment(
        file.filename,
        { ...toolUseContext, fileReadingLimits: { maxTokens: POST_COMPACT_MAX_TOKENS_PER_FILE } },
        'tengu_post_compact_file_restore_success',
        'tengu_post_compact_file_restore_error',
        'compact',
      )
      return attachment ? createAttachmentMessage(attachment) : null
    }),
  )
  
  // 4. 按 token 预算过滤
  let usedTokens = 0
  return results.filter((result): result is AttachmentMessage => {
    if (result === null) return false
    const attachmentTokens = roughTokenCountEstimation(jsonStringify(result))
    if (usedTokens + attachmentTokens <= POST_COMPACT_TOKEN_BUDGET) {
      usedTokens += attachmentTokens
      return true
    }
    return false
  })
}
```

### 5.3 排除规则

```typescript
function shouldExcludeFromPostCompactRestore(
  filename: string,
  agentId?: AgentId,
): boolean {
  const normalizedFilename = expandPath(filename)
  
  // 排除计划文件
  const planFilePath = expandPath(getPlanFilePath(agentId))
  if (normalizedFilename === planFilePath) return true
  
  // 排除所有类型的 CLAUDE.md 文件
  const normalizedMemoryPaths = new Set(
    MEMORY_TYPE_VALUES.map(type => expandPath(getMemoryPath(type))),
  )
  if (normalizedMemoryPaths.has(normalizedFilename)) return true
  
  return false
}
```

### 5.4 Preserved Messages 去重

**问题**：在部分压缩中，`messagesToKeep` 已经包含了一些 Read tool 的结果。如果恢复的文件与这些结果重复，就是纯粹的浪费（单次压缩可能浪费高达 25K tokens）。

**解决方案**：diff against preserved。

```typescript
function collectReadToolFilePaths(messages: Message[]): Set<string> {
  const stubIds = new Set<string>()
  
  // 先收集 dedup stub 的 tool_use_id
  for (const message of messages) {
    if (message.type !== 'user' || !Array.isArray(message.message.content)) continue
    for (const block of message.message.content) {
      if (
        block.type === 'tool_result' &&
        typeof block.content === 'string' &&
        block.content.startsWith(FILE_UNCHANGED_STUB)
      ) {
        stubIds.add(block.tool_use_id)
      }
    }
  }
  
  // 收集非 stub 的 Read tool 的 file_path
  const paths = new Set<string>()
  for (const message of messages) {
    if (message.type !== 'assistant' || !Array.isArray(message.message.content)) continue
    for (const block of message.message.content) {
      if (
        block.type !== 'tool_use' ||
        block.name !== FILE_READ_TOOL_NAME ||
        stubIds.has(block.id)
      ) {
        continue
      }
      const input = block.input
      if (
        input &&
        typeof input === 'object' &&
        'file_path' in input &&
        typeof input.file_path === 'string'
      ) {
        paths.add(expandPath(input.file_path))
      }
    }
  }
  return paths
}
```

## 6. Skill 恢复

### 6.1 为什么需要恢复 Skills

Skill 文件通常较大（verify 约 18.7KB，claude-api 约 20.1KB）。压缩会丢弃技能调用的上下文，需要恢复以便模型继续遵循技能指南。

### 6.2 恢复策略

```typescript
export function createSkillAttachmentIfNeeded(
  agentId?: string,
): AttachmentMessage | null {
  const invokedSkills = getInvokedSkillsForAgent(agentId)
  
  if (invokedSkills.size === 0) return null
  
  // 按调用时间排序（最近优先）
  let usedTokens = 0
  const skills = Array.from(invokedSkills.values())
    .sort((a, b) => b.invokedAt - a.invokedAt)
    .map(skill => ({
      name: skill.skillName,
      path: skill.skillPath,
      content: truncateToTokens(
        skill.content,
        POST_COMPACT_MAX_TOKENS_PER_SKILL,  // 5K per skill
      ),
    }))
    .filter(skill => {
      const tokens = roughTokenCountEstimation(skill.content)
      if (usedTokens + tokens > POST_COMPACT_SKILLS_TOKEN_BUDGET) {  // 25K total
        return false
      }
      usedTokens += tokens
      return true
    })
  
  if (skills.length === 0) return null
  
  return createAttachmentMessage({
    type: 'invoked_skills',
    skills,
  })
}
```

### 6.3 Per-Skill 截断

```typescript
const SKILL_TRUNCATION_MARKER =
  '\n\n[... skill content truncated for compaction; use Read on the skill path if you need the full text]'

function truncateToTokens(content: string, maxTokens: number): string {
  if (roughTokenCountEstimation(content) <= maxTokens) {
    return content
  }
  const charBudget = maxTokens * 4 - SKILL_TRUNCATION_MARKER.length
  return content.slice(0, charBudget) + SKILL_TRUNCATION_MARKER
}
```

**Rationale**：
- 保留每个 skill 的头部（通常是设置和使用说明）
- 添加 marker 告知模型可以 Read 完整文件
- 按调用时间排序，最近的 skill 优先保留

### 6.4 不重置 sentSkillNames

```typescript
// Intentionally NOT resetting sentSkillNames
// 理由：重新注入完整的 skill_listing (~4K tokens) 每次压缩都是纯粹的 cache_creation，
// 边际效益低。模型仍然有 SkillTool schema 和 invoked_skills attachment。
```

## 7. Prompt Too Long 重试机制

### 7.1 问题

压缩请求本身也可能触发 prompt-too-long 错误，尤其是在上下文非常大的情况下。

### 7.2 重试策略

```typescript
const MAX_PTL_RETRIES = 3
const PTL_RETRY_MARKER = '[earlier conversation truncated for compaction retry]'

let ptlAttempts = 0
for (;;) {
  summaryResponse = await streamCompactSummary({ ... })
  summary = getAssistantMessageText(summaryResponse)
  
  if (!summary?.startsWith(PROMPT_TOO_LONG_ERROR_MESSAGE)) break
  
  ptlAttempts++
  const truncated =
    ptlAttempts <= MAX_PTL_RETRIES
      ? truncateHeadForPTLRetry(messagesToSummarize, summaryResponse)
      : null
  
  if (!truncated) {
    logEvent('tengu_compact_failed', { reason: 'prompt_too_long', ... })
    throw new Error(ERROR_MESSAGE_PROMPT_TOO_LONG)
  }
  
  messagesToSummarize = truncated
  retryCacheSafeParams = {
    ...retryCacheSafeParams,
    forkContextMessages: truncated,
  }
}
```

### 7.3 截断逻辑

```typescript
export function truncateHeadForPTLRetry(
  messages: Message[],
  ptlResponse: AssistantMessage,
): Message[] | null {
  // 移除之前的 synthetic marker
  const input =
    messages[0]?.type === 'user' &&
    messages[0].isMeta &&
    messages[0].message.content === PTL_RETRY_MARKER
      ? messages.slice(1)
      : messages
  
  const groups = groupMessagesByApiRound(input)
  if (groups.length < 2) return null
  
  // 计算 token gap
  const tokenGap = getPromptTooLongTokenGap(ptlResponse)
  let dropCount: number
  
  if (tokenGap !== undefined) {
    // 有明确 gap：累加直到覆盖
    let acc = 0
    dropCount = 0
    for (const g of groups) {
      acc += roughTokenCountEstimationForMessages(g)
      dropCount++
      if (acc >= tokenGap) break
    }
  } else {
    // 无法解析 gap：回退到丢弃 20%
    dropCount = Math.max(1, Math.floor(groups.length * 0.2))
  }
  
  // 至少保留一组
  dropCount = Math.min(dropCount, groups.length - 1)
  if (dropCount < 1) return null
  
  const sliced = groups.slice(dropCount).flat()
  
  // 如果第一组被丢弃，prepend synthetic marker（API 要求第一条是 user）
  if (sliced[0]?.type === 'assistant') {
    return [
      createUserMessage({ content: PTL_RETRY_MARKER, isMeta: true }),
      ...sliced,
    ]
  }
  return sliced
}
```

## 8. 图像剥离

### 8.1 原因

用户消息可能包含图像附件。压缩时：
- 图像对生成摘要没有帮助
- 图像会导致压缩请求本身触发 prompt-too-long
- CCD 会话中用户频繁附加图像

### 8.2 实现

```typescript
export function stripImagesFromMessages(messages: Message[]): Message[] {
  return messages.map(message => {
    if (message.type !== 'user') return message
    
    const content = message.message.content
    if (!Array.isArray(content)) return message
    
    let hasMediaBlock = false
    const newContent = content.flatMap(block => {
      if (block.type === 'image') {
        hasMediaBlock = true
        return [{ type: 'text' as const, text: '[image]' }]
      }
      if (block.type === 'document') {
        hasMediaBlock = true
        return [{ type: 'text' as const, text: '[document]' }]
      }
      // 处理 tool_result 内层的图像
      if (block.type === 'tool_result' && Array.isArray(block.content)) {
        let toolHasMedia = false
        const newToolContent = block.content.map(item => {
          if (item.type === 'image') {
            toolHasMedia = true
            return { type: 'text' as const, text: '[image]' }
          }
          if (item.type === 'document') {
            toolHasMedia = true
            return { type: 'text' as const, text: '[document]' }
          }
          return item
        })
        if (toolHasMedia) {
          hasMediaBlock = true
          return [{ ...block, content: newToolContent }]
        }
      }
      return [block]
    })
    
    if (!hasMediaBlock) return message
    
    return {
      ...message,
      message: { ...message.message, content: newContent },
    }
  })
}
```

## 9. 重新注入的 Attachments 剥离

### 9.1 问题

`skill_discovery`/`skill_listing` attachments 会在压缩后通过 `resetSentSkillNames()` 和下次 turn 的 discovery signal 重新注入。如果在压缩时保留它们：
- 浪费 tokens
- 污染摘要（包含过时的 skill 建议）

### 9.2 实现

```typescript
export function stripReinjectedAttachments(messages: Message[]): Message[] {
  if (feature('EXPERIMENTAL_SKILL_SEARCH')) {
    return messages.filter(
      m =>
        !(
          m.type === 'attachment' &&
          (m.attachment.type === 'skill_discovery' ||
            m.attachment.type === 'skill_listing')
        ),
    )
  }
  return messages
}
```

## 10. 异步 Agent 状态保留

### 10.1 场景

压缩时可能有异步 agent：
- 仍在后台运行（需要告知模型不要 spawn 重复的）
- 已完成但结果尚未检索（需要保留结果）

### 10.2 实现

```typescript
export async function createAsyncAgentAttachmentsIfNeeded(
  context: ToolUseContext,
): Promise<AttachmentMessage[]> {
  const appState = context.getAppState()
  const asyncAgents = Object.values(appState.tasks).filter(
    (task): task is LocalAgentTaskState => task.type === 'local_agent',
  )
  
  return asyncAgents.flatMap(agent => {
    if (
      agent.retrieved ||
      agent.status === 'pending' ||
      agent.agentId === context.agentId
    ) {
      return []
    }
    return [
      createAttachmentMessage({
        type: 'task_status',
        taskId: agent.agentId,
        taskType: 'local_agent',
        description: agent.description,
        status: agent.status,
        deltaSummary:
          agent.status === 'running'
            ? (agent.progress?.summary ?? null)
            : (agent.error ?? null),
        outputFilePath: getTaskOutputPath(agent.agentId),
      }),
    ]
  })
}
```

## 11. Plan Mode 指令保留

### 11.1 问题

Plan mode 指令通常只在 tool-use turn 通过 `getAttachmentMessages` 注入。压缩会丢失这些指令，导致模型忘记自己处于 plan mode。

### 11.2 实现

```typescript
export async function createPlanModeAttachmentIfNeeded(
  context: ToolUseContext,
): Promise<AttachmentMessage | null> {
  const appState = context.getAppState()
  if (appState.toolPermissionContext.mode !== 'plan') {
    return null
  }
  
  const planFilePath = getPlanFilePath(context.agentId)
  const planExists = getPlan(context.agentId) !== null
  
  return createAttachmentMessage({
    type: 'plan_mode',
    reminderType: 'full',
    isSubAgent: !!context.agentId,
    planFilePath,
    planExists,
  })
}
```

## 13. 总结

会话压缩机制是 Claude Code 上下文管理的核心，通过以下策略实现高效的上下文控制：

| 策略 | 描述 |
|------|------|
| 分层压缩 | Full/Partial、Auto/Manual 多种模式 |
| Token 预算 | 文件恢复 50K、Skills 25K、单文件/单 skill 5K |
| 状态恢复 | 文件、Skills、Plan、Async Agents、Plan Mode |
| 去重优化 | Preserved messages diff、sentSkillNames 不重置 |
| 重试机制 | Prompt too long 时截断头部重试（最多 3 次） |
| 内容剥离 | 图像、重新注入的 attachments |
| Cache 优化 | Forked agent 复用 prompt cache |
| Telemetry | 详细的 token 统计和上下文分析 |

这套机制确保了在有限 token 预算内，模型始终拥有最相关的上下文信息，同时最小化压缩带来的信息损失。
