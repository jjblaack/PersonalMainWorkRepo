## 1. 架构设计总结

### 1.1 分层架构回顾

Claude Code 采用清晰的分层架构：

```
┌─────────────────────────────────────────────────────────────┐
│  入口层 (Entry Layer)                                        │
│  - main.tsx: CLI 解析，React 初始化                           │
│  - commands.ts: 命令注册                                     │
│  - entrypoints/: 初始化逻辑                                  │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  命令层 (Command Layer)                                      │
│  - 用户直接调用的接口 (/command)                            │
│  - ~88 个命令实现                                             │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  工具层 (Tool Layer)                                         │
│  - AI 可调用的能力                                           │
│  - ~44 个工具实现                                             │
│  - Tool.ts: 工具基类和接口                                   │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  服务层 (Service Layer)                                      │
│  - 外部 API 封装 (Anthropic API, MCP, LSP, OAuth)            │
│  - 业务能力 (压缩、记忆、估算)                               │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  基础设施层 (Infrastructure)                                 │
│  - types/: 类型定义                                          │
│  - utils/: 工具函数                                          │
│  - schemas/: Zod Schema                                     │
│  - constants/: 常量                                         │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 模块划分原则

1. **单一职责**: 每个模块有明确的职责
2. **依赖方向**: 依赖只能向下，不能向上
3. **接口隔离**: 通过接口通信，隐藏实现细节
4. **关注点分离**: UI、业务逻辑、数据访问分离

## 2. 核心设计模式

### 2.1 策略模式 (Strategy Pattern)

**应用场景**: 工具系统

```typescript
// 每个工具是一个独立的策略
interface Tool {
  execute(input, context): Promise<ToolResult>;
}

class BashTool implements Tool { ... }
class FileReadTool implements Tool { ... }
class GrepTool implements Tool { ... }

// 上下文使用策略
function executeTool(tool: Tool, input, context) {
  return tool.execute(input, context);
}
```

**优点**:
- 易于添加新工具
- 工具之间解耦
- 可独立测试

### 2.2 工厂模式 (Factory Pattern)

**应用场景**: 工具注册、命令注册

```typescript
// 简单工厂
function getTools(): Tool[] {
  return [
    new AgentTool(),
    new BashTool(),
    new FileReadTool(),
    // ...
  ];
}

function findTool(name: string): Tool | undefined {
  return getTools().find(t => t.name === name);
}
```

**优点**:
- 集中管理对象创建
- 便于条件加载
- 易于扩展

### 2.3 观察者模式 (Observer Pattern)

**应用场景**: 进度反馈、状态更新

```typescript
// 主题 (被观察者)
class QueryEngine {
  private observers: ((progress) => void)[] = [];
  
  subscribe(callback) {
    this.observers.push(callback);
  }
  
  notify(progress) {
    this.observers.forEach(cb => cb(progress));
  }
}

// 观察者 (UI 组件)
engine.subscribe((progress) => {
  updateProgressBar(progress);
});
```

**优点**:
- 松耦合
- 支持多个观察者
- 动态订阅/取消

### 2.4 责任链模式 (Chain of Responsibility)

**应用场景**: 工具执行流程、权限检查

```
请求 → 查找工具 → 验证输入 → 权限检查 → [用户审批] → 执行 → 返回结果
```

**优点**:
- 每个环节独立
- 易于添加新环节
- 灵活的流程控制

### 2.5 代理模式 (Proxy Pattern)

**应用场景**: AgentTool spawned 子代理

```typescript
class AgentTool extends Tool {
  async execute(input, context) {
    // 创建代理 (子代理会话)
    const agent = await createAgentSession(input);
    
    // 通过代理执行
    const result = await agent.run();
    
    // 返回结果
    return summarize(result);
  }
}
```

**优点**:
- 隔离子代理
- 控制访问
- 添加额外逻辑 (进度跟踪)

### 2.6 组合模式 (Composite Pattern)

**应用场景**: UI 组件树

```tsx
// 组件组合
function App() {
  return (
    <FullscreenLayout>
      <Header />
      <MessageList>
        <UserMessage />
        <AssistantMessage />
        <ToolUseMessage />
      </MessageList>
      <InputArea />
      <StatusBar />
    </FullscreenLayout>
  );
}
```

**优点**:
- 树形结构
- 统一接口
- 递归渲染

### 3.3 错误处理模式

```typescript
// 统一错误处理
async function withErrorHandling<T>(
  fn: () => Promise<T>,
  context: string
): Promise<T> {
  try {
    return await fn();
  } catch (error) {
    logError(error, context);
    
    if (error instanceof APIError) {
      throw new UserFriendlyError(formatAPIError(error));
    }
    
    throw error;
  }
}

// 错误分类处理
function handleToolError(error: unknown, toolName: string) {
  if (isPermissionDenied(error)) {
    return { success: false, error: 'Permission denied' };
  }
  if (isTimeout(error)) {
    return { success: false, error: 'Timeout' };
  }
  return { success: false, error: errorMessage(error) };
}
```

## 4. 性能优化技术

### 4.1 并行预取

```typescript
// 启动时并行预取
profileCheckpoint('main_tsx_entry');
startMdmRawRead();        // MDM 设置读取 (~65ms)
startKeychainPrefetch();  // Keychain 读取 (~65ms)

// 与模块加载 (~135ms) 并行
// 总计：~135ms (而非 135+65+65=265ms)
```

**收益**: 节省约 130ms 启动时间

### 4.2 惰性加载

```typescript
// 重模块按需加载
const initializeTelemetry = async () => {
  const { initializeTelemetry } = await import('./telemetry.js');
  return initializeTelemetry();
};

// 仅在需要时调用
if (options.telemetry) {
  await initializeTelemetry();
}
```

**收益**: 减少初始加载负担 ~1MB

### 4.3 特性标志死代码消除

```typescript
import { feature } from 'bun:bundle';

// 构建时消除
const voiceModule = feature('VOICE_MODE')
  ? require('./voice.js')
  : null;

// 如果 VOICE_MODE 未启用，voice 代码完全不打包
```

**收益**: 减少打包体积 10-20%

### 4.4 代码分割

```typescript
// 动态 import 实现代码分割
const FullscreenLayout = async () => {
  const { FullscreenLayout } = await import('./components/FullscreenLayout.js');
  return <FullscreenLayout />;
};
```

**收益**: 初始加载更快，按需加载

### 4.5 Memoize 缓存

```typescript
// memoize 确保只执行一次
export const init = memoize(async (): Promise<void> => {
  // 初始化逻辑
});

// 后续调用返回缓存结果
await init();  // 执行
await init();  // 返回缓存
```

**收益**: 避免重复初始化

## 5. 安全设计

### 5.1 权限模型

```
权限级别: low | medium | high

工具分类:
- low: FileRead, Glob, Grep (自动执行)
- medium: FileEdit, WebFetch (需要审批)
- high: Bash, FileWrite, Agent (需要审批)

权限模式:
- default: 所有操作需要审批
- auto: 低风险自动，高风险审批
- bypassPermissions: 开发模式 (仅安全环境)
```

### 5.2 边界检查

```typescript
// 文件系统边界
function isPathAllowed(filePath: string, allowedPaths: string[]): boolean {
  return allowedPaths.some(p => filePath.startsWith(p));
}

// 命令边界
const BLOCKED_COMMANDS = ['rm -rf /', 'mkfs', 'dd', 'chmod -R 777'];

// 网络边界
const BLOCKED_HOSTS = ['localhost', '169.254.169.254'];
```

### 5.3 认证机制

```typescript
// OAuth 2.0 + PKCE
async function startAuthFlow() {
  const state = generateRandomState();
  const codeVerifier = generateCodeVerifier();
  
  // 存储 PKCE 参数
  await secureStorage.set('state', state);
  await secureStorage.set('verifier', codeVerifier);
  
  // 构建授权 URL
  return buildAuthUrl(state, codeVerifier);
}

// JWT 会话认证
async function verifySessionToken(token: string) {
  const { payload } = await jwtVerify(token, secret);
  return payload;
}
```
