# Harness 工程专题 - 09-设计思想总结

本文总结 Harness 工程的核心设计思想和可借鉴模式。

## 1. 防御深度 (Defense in Depth)

Harness 系统采用多层防御策略，确保单层绕过不会导致安全问题。

### 1.1 Bash 安全检查层次

```
┌─────────────────────────────────────────────────────────────┐
│ Layer 1: 规则匹配                                            │
│ - 工具级 deny/ask/allow 规则                                   │
│ - 内容级 deny/ask/allow 规则                                   │
│ - 环境变量剥离、包装器剥离                                    │
└─────────────────────────────────────────────────────────────┘
                              │ 通过
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ Layer 2: AST 语义分析                                         │
│ - tree-sitter 解析                                             │
│ - 危险节点类型检测 (eval, subshell, etc.)                     │
│ - 控制流检测 (for, while, if)                                 │
└─────────────────────────────────────────────────────────────┘
                              │ 通过
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ Layer 3: 路径约束                                            │
│ - 工作目录检查                                                │
│ - 敏感路径保护 (.git, .claude, .vscode)                       │
│ - 输出重定向检查                                              │
└─────────────────────────────────────────────────────────────┘
                              │ 通过
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ Layer 4: 命令注入检测                                         │
│ - 命令替换检测 $(...)                                         │
│ - 进程替换检测 <(...) >(...)                                  │
│ - Shell 操作符检测 && || ; | &                                │
└─────────────────────────────────────────────────────────────┘
                              │ 通过
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ Layer 5: Sandbox 隔离                                         │
│ - 文件系统隔离 (bubblewrap)                                   │
│ - 网络隔离 (域名白名单)                                        │
│ - 进程隔离                                                    │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 可借鉴模式

```typescript
// 模式：多层验证管道
async function validateWithDepth(input: T): Promise<Result> {
  // Layer 1: 快速规则检查
  const ruleResult = await checkRules(input)
  if (!ruleResult.pass) return ruleResult
  
  // Layer 2: 深度结构分析
  const astResult = await analyzeAST(input)
  if (!astResult.pass) return astResult
  
  // Layer 3: 上下文约束
  const contextResult = await checkContext(input)
  if (!contextResult.pass) return contextResult
  
  // Layer 4: 运行时保护
  return await executeWithSandbox(input)
}
```

## 2. 失败关闭 (Fail-Closed)

当系统无法确定安全性时，默认选择保守策略。

### 2.1 实现示例

```typescript
// AST 解析失败 → too-complex → ask
if (result.kind === 'too-complex') {
  return { kind: 'too-complex', reason: '无法静态分析此命令' }
}

// 规则匹配不确定 → ask
if (noRulesMatch) {
  return { behavior: 'ask' }
}

// 分类器 API 错误 → fail-closed (默认)
if (classifierResult.unavailable && ironGateClosed) {
  return { behavior: 'deny', reason: '分类器不可用' }
}
```

### 2.2 可借鉴模式

```typescript
// 模式：保守默认
function decideWithUncertainty(info: Partial<Info>): Decision {
  if (info.complete) {
    return analyze(info)
  }
  // 信息不足 → 保守决策
  return { action: 'deny', reason: '信息不足，需要人工审批' }
}

// 模式：配置化失败策略
enum FailStrategy {
  FailClosed = 'deny_all',
  FailOpen = 'allow_all',
  FailPrompt = 'ask_user',
}

function handleFailure(
  error: Error,
  strategy: FailStrategy,
): Decision {
  switch (strategy) {
    case FailStrategy.FailClosed:
      return { action: 'deny', reason: error.message }
    case FailStrategy.FailOpen:
      return { action: 'allow', reason: '故障放行' }
    case FailStrategy.FailPrompt:
      return { action: 'ask', reason: error.message }
  }
}
```

## 3. 渐进式自动化 (Progressive Automation)

从完全手动到完全自动，系统提供多个自动化级别。

│ - 默认模式                                                   │
└────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌────────────────────────────────────────────────────────────┐
│ Level 1: Hook 自动化                                        │
│ - 本地脚本自动决策                                           │
│ - 快速、确定性的规则                                         │
└────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌────────────────────────────────────────────────────────────┐
│ Level 2: Bash 分类器                                        │
│ - 基于 prompt 规则匹配                                        │

└────────────────────────────────────────────────────────────┘
                              │

┌────────────────────────────────────────────────────────────┐
│ Level 3: Auto 模式分类器                                    │

                              ▼

function getDecision(

    case 'full':

  }
}

## 4. 原子竞争 (Atomic Race)

多个决策源并行运行，第一个有效决策获胜。

### 4.1 实现

```typescript
function createResolveOnce<T>(
  resolve: (value: T) => void,
): { resolve: (v: T) => void; claim: () => boolean; isResolved: () => boolean } {
  let resolved = false
  
  return {
    resolve: (value: T) => {
      if (!resolved) {
        resolved = true
        resolve(value)
      }
    },
    claim: () => {
      if (resolved) return false
      resolved = true
      return true
    },
    isResolved: () => resolved,
  }
}

// 使用示例
const { claim, resolve: resolveOnce } = createResolveOnce(resolve)

// 并行源 1：用户交互
onUserAllow() {
  if (!claim()) return  // 已被其他源处理
  resolveOnce(buildAllow())
}

// 并行源 2：Hook
const hookDecision = await runHooks()
if (hookDecision && claim()) {
  resolveOnce(hookDecision)
}

// 并行源 3：分类器
const classifierResult = await runClassifier()
if (classifierResult.shouldAllow && claim()) {
  resolveOnce(buildAllow())
}
```

### 4.2 可借鉴模式

```typescript
// 模式：并行决策竞赛
async function decisionRace<T>(
  sources: DecisionSource<T>[],
): Promise<T> {
  const { claim, resolve: resolveOnce } = createResolveOnce()
  
  const promises = sources.map(async source => {
    const result = await source.run()
    if (result && claim()) {
      resolveOnce(result)
    }
  })
  
  return new Promise<T>((resolve) => {
    // resolveOnce 会被第一个成功的源调用
  })
}
```

## 5. 优雅降级 (Graceful Degradation)

当高级功能不可用时，系统回退到基础功能。

### 5.1 实现示例

```typescript
// 分类器不可用 → 回退到用户询问
if (classifierResult.unavailable) {
  if (ironGateClosed) {
    return { behavior: 'deny' }  // Fail closed
  }
  return result  // Fall through to user prompt
}

// AST 不可用 → 回退到正则检测
if (!astParseSucceeded) {
  const safetyResult = await bashCommandIsSafeAsync(command)
  if (safetyResult.behavior !== 'passthrough') {
    return safetyResult
  }
}

// tree-sitter 加载失败 → 回退到 shell-quote
if (!mod) {
  return null  // 调用者使用 legacy 路径
}
```

### 5.2 可借鉴模式

```typescript
// 模式：功能降级链
async function processWithFallback(input: T): Promise<Result> {
  // 尝试高级方法
  try {
    const result = await advancedMethod(input)
    if (result.success) return result
  } catch (e) {
    logWarning('Advanced method failed, trying fallback', e)
  }
  
  // 尝试中级方法
  try {
    const result = await intermediateMethod(input)
    if (result.success) return result
  } catch (e) {
    logWarning('Intermediate method failed, trying basic fallback', e)
  }
  
  // 基础方法（总是可用）
  return basicMethod(input)
}
```

## 6. 预检优化 (Speculative Check)

提前启动耗时操作，与 UI 并行运行，实现零延迟体验。

### 6.1 实现

```typescript
// 用户看到对话框时，分类器已在后台运行
function startSpeculativeClassifierCheck(
  command: string,
  signal: AbortSignal,
): void {
  const promise = classifyBashCommand(command, ..., signal)
  promise.catch(() => {})  // 防止未处理 rejection
  speculativeChecks.set(command, promise)
}

// 用户交互时检查分类器是否已完成
async function executeAsyncClassifierCheck(..., callbacks: Callbacks) {
  const classifierResult = await speculativeChecks.get(command)
  
  if (!callbacks.shouldContinue()) return  // 用户已决策
  
  if (classifierResult.matches && classifierResult.confidence === 'high') {
    callbacks.onAllow({
      type: 'classifier',
      reason: `Allowed by rule: "${classifierResult.matchedDescription}"`,
    })
  }
}
```

### 6.2 可借鉴模式

```typescript
// 模式：预检 + 宽限期
class SpeculativeExecutor<T> {
  private pendingChecks = new Map<string, Promise<T>>()
  
  startSpeculative(key: string, compute: () => Promise<T>): void {
    const promise = compute()
    promise.catch(() => {})
    this.pendingChecks.set(key, promise)
  }
  
  async getResult(
    key: string,
    gracePeriodMs: number,
    shouldContinue: () => boolean,
  ): Promise<T | null> {
    const promise = this.pendingChecks.get(key)
    if (!promise) return null
    
    // 宽限期内等待结果
    const result = await Promise.race([
      promise,
      new Promise<null>(resolve => 
        setTimeout(() => resolve(null), gracePeriodMs)
      ),
    ])
    
    if (result && shouldContinue()) {
      return result
    }
    
    return null
  }
}
```

## 7. 宽限期设计 (Grace Period)

防止用户意外操作取消后台任务。

### 7.1 实现

```typescript
const GRACE_PERIOD_MS = 200

onUserInteraction() {
  if (Date.now() - permissionPromptStartTimeMs < GRACE_PERIOD_MS) {
    return  // 忽略早期交互
  }
  userInteracted = true
  clearClassifierChecking(ctx.toolUseID)
}
```

### 7.2 可借鉴模式

```typescript
// 模式：可配置的宽限期
function withGracePeriod<T>(
  handler: () => T,
  gracePeriodMs: number,
  startTime: number,
): T | undefined {
  if (Date.now() - startTime < gracePeriodMs) {
    return undefined  // 忽略
  }
  return handler()
}

// 使用
onInteraction() {
  const result = withGracePeriod(
    () => cancelBackgroundTask(),
    200,
    promptStartTime,
  )
  if (result !== undefined) {
    // 处理取消
  }
}
```

### 9.2 可借鉴模式

```typescript
// 模式：配置订阅者
class ConfigManager<T> {
  private config: T
  private subscribers = new Set<() => void>()
  
  subscribe(callback: () => void): () => void {
    this.subscribers.add(callback)
    return () => this.subscribers.delete(callback)
  }
  
  update(partial: Partial<T>): void {
    this.config = { ...this.config, ...partial }
    this.subscribers.forEach(cb => cb())
  }
  
  get(): T {
    return this.config
  }
}
```

## 10. 来源优先级 (Source Priority)

不同来源的配置按优先级合并。

### 10.1 实现

```typescript
const PERMISSION_RULE_SOURCES = [
  'policySettings',   // 最高优先级
  'flagSettings',
  'userSettings',
  'projectSettings',
  'localSettings',
  'cliArg',
  'session',          // 最低优先级
] as const

function getRules(
  context: ToolPermissionContext,
  behavior: PermissionBehavior,
): PermissionRule[] {
  const rules = PERMISSION_RULE_SOURCES.flatMap(source =>
    (context[`${behavior}Rules`][source] || []).map(ruleString => ({
      source,
      ruleBehavior: behavior,
      ruleValue: permissionRuleValueFromString(ruleString),
    })),
  )
  return rules
}
```

### 10.2 可借鉴模式

```typescript
// 模式：优先级配置合并
const SOURCE_PRIORITY = ['global', 'org', 'team', 'project', 'local'] as const

function getMergedConfig<T>(
  sources: Record<string, Partial<T>>,
): T {
  let merged = {} as T
  
  for (const source of SOURCE_PRIORITY) {
    merged = { ...merged, ...sources[source] }  // 高优先级覆盖低优先级
  }
  
  return merged
}
```

## 11. 总结

Harness 工程的核心设计原则：

| 原则 | 应用场景 | 价值 |
|------|----------|------|
| **防御深度** | 安全检查 | 单层绕过不导致安全问题 |
| **失败关闭** | 不确定决策 | 安全优先于便利 |
| **渐进式自动化** | 权限决策 | 用户可选择自动化级别 |
| **原子竞争** | 多决策源 | 避免重复处理 |
| **优雅降级** | 功能不可用 | 保持基本功能可用 |
| **预检优化** | 耗时操作 | 零延迟用户体验 |
| **宽限期** | 用户交互 | 防止意外取消 |
| **可解释性** | 审计调试 | 每个决策可追溯 |
| **热更新** | 配置变化 | 无需重启生效 |
| **来源优先级** | 配置合并 | 灵活的管理层次 |

这些设计思想不仅适用于 AI 控制系统，也适用于任何需要安全、灵活、可解释的权限管理系统。
