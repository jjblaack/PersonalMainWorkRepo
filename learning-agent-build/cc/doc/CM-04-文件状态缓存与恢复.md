# CM-04 - 文件状态缓存与恢复

## 1. 概述

文件状态缓存（`readFileState`）是 Claude Code 用于追踪最近读取文件内容的内存缓存。它在会话压缩、跨会话恢复、工具调用优化等场景中发挥关键作用。

### 1.1 核心功能

1. **最近读取追踪**：记录用户最近读取的文件内容和时间戳
2. **压缩后恢复**：会话压缩后自动恢复最近读取的文件
3. **状态外置**：大文件内容存储在缓存中，而非直接放在上下文中
4. **变更检测**：通过 `contentDiffersFromDisk` 检测文件是否被修改

## 3. 压缩后文件恢复

### 3.1 恢复流程

```typescript
export async function createPostCompactFileAttachments(
  readFileState: Record<string, { content: string; timestamp: number }>,
  toolUseContext: ToolUseContext,
  maxFiles: number = POST_COMPACT_MAX_FILES_TO_RESTORE,
  preservedMessages: Message[] = [],
): Promise<AttachmentMessage[]> {
  // 1. 收集 preservedMessages 中已有的 Read 结果
  const preservedReadPaths = collectReadToolFilePaths(preservedMessages)
  
  // 2. 选择最近读取的文件（排除已存在和排除列表）
  const recentFiles = Object.entries(readFileState)
    .map(([filename, state]) => ({ filename, ...state }))
    .filter(file =>
      !shouldExcludeFromPostCompactRestore(file.filename, toolUseContext.agentId) &&
      !preservedReadPaths.has(expandPath(file.filename)),
    )
    .sort((a, b) => b.timestamp - a.timestamp)
    .slice(0, maxFiles)
  
  // 3. 并行读取文件（使用 FileReadTool 获取新鲜内容）
  const results = await Promise.all(
    recentFiles.map(async file => {
      const attachment = await generateFileAttachment(
        file.filename,
        { ...toolUseContext, fileReadingLimits: { maxTokens: POST_COMPACT_MAX_TOKENS_PER_FILE } },
        'tengu_post_compact_file_restore_success',
        'tengu_post_compact_file_restore_error',
        'compact',
      )
      return attachment ? createAttachmentMessage(attachment) : null
    }),
  )
  
  // 4. 按 token 预算过滤
  let usedTokens = 0
  return results.filter((result): result is AttachmentMessage => {
    if (result === null) return false
    const attachmentTokens = roughTokenCountEstimation(jsonStringify(result))
    if (usedTokens + attachmentTokens <= POST_COMPACT_TOKEN_BUDGET) {
      usedTokens += attachmentTokens
      return true
    }
    return false
  })
}
```

### 3.2 排除规则

```typescript
function shouldExcludeFromPostCompactRestore(
  filename: string,
  agentId?: AgentId,
): boolean {
  const normalizedFilename = expandPath(filename)
  
  // 排除计划文件
  try {
    const planFilePath = expandPath(getPlanFilePath(agentId))
    if (normalizedFilename === planFilePath) return true
  } catch {
    // 无法获取计划路径时继续
  }
  
  // 排除所有类型的 CLAUDE.md 文件
  try {
    const normalizedMemoryPaths = new Set(
      MEMORY_TYPE_VALUES.map(type => expandPath(getMemoryPath(type))),
    )
    if (normalizedMemoryPaths.has(normalizedFilename)) return true
  } catch {
    // 无法获取记忆路径时继续
  }
  
  return false
}
```

**排除的文件类型**：
- 计划文件（单独通过 plan attachment 恢复）
- CLAUDE.md 文件（通过 CLAUDE.md 系统注入）
- 已在 preservedMessages 中的 Read 结果（去重）

### 3.3 Preserved Messages 去重

**问题**：在部分压缩中，`messagesToKeep` 已经包含了一些 Read tool 的结果。如果恢复的文件与这些结果重复，就是纯粹的浪费。

**解决方案**：收集 preservedMessages 中的 Read tool 路径。

```typescript
function collectReadToolFilePaths(messages: Message[]): Set<string> {
  const stubIds = new Set<string>()
  
  // 先收集 dedup stub 的 tool_use_id
  for (const message of messages) {
    if (message.type !== 'user' || !Array.isArray(message.message.content)) continue
    for (const block of message.message.content) {
      if (
        block.type === 'tool_result' &&
        typeof block.content === 'string' &&
        block.content.startsWith(FILE_UNCHANGED_STUB)
      ) {
        stubIds.add(block.tool_use_id)
      }
    }
  }
  
  // 收集非 stub 的 Read tool 的 file_path
  const paths = new Set<string>()
  for (const message of messages) {
    if (message.type !== 'assistant' || !Array.isArray(message.message.content)) continue
    for (const block of message.message.content) {
      if (
        block.type !== 'tool_use' ||
        block.name !== FILE_READ_TOOL_NAME ||
        stubIds.has(block.id)
      ) {
        continue
      }
      const input = block.input
      if (
        input &&
        typeof input === 'object' &&
        'file_path' in input &&
        typeof input.file_path === 'string'
      ) {
        paths.add(expandPath(input.file_path))
      }
    }
  }
  return paths
}
```

## 4. 内容差异化检测

### 4.1 设计目的

当 CLAUDE.md 文件内容经过处理（移除 frontmatter、移除注释）后，与磁盘上的原始内容不同，系统需要：

1. 记录这种差异
2. 保留原始内容用于缓存去重
3. 在 Edit/Write 操作前要求显式 Read

### 4.2 MemoryFileInfo 结构

```typescript
type MemoryFileInfo = {
  path: string
  type: MemoryType
  content: string          // 处理后的内容（注入上下文）
  parent?: string          // 包含此文件的父文件路径
  globs?: string[]         // 条件规则的路径模式
  contentDiffersFromDisk?: boolean  // 内容是否与磁盘不同
  rawContent?: string      // 磁盘原始内容（当不同时）
}
```

### 4.3 处理逻辑

```typescript
function parseMemoryFileContent(
  rawContent: string,
  filePath: string,
  type: MemoryType,
  includeBasePath?: string,
): { info: MemoryFileInfo | null; includePaths: string[] } {
  // 1. 解析 frontmatter
  const { content: withoutFrontmatter, paths } = parseFrontmatterPaths(rawContent)
  
  // 2. 移除 HTML 注释
  const strippedContent = hasComment && tokens
    ? stripHtmlCommentsFromTokens(tokens).content
    : withoutFrontmatter
  
  // 3. AutoMem/TeamMem 截断
  let finalContent = strippedContent
  if (type === 'AutoMem' || type === 'TeamMem') {
    finalContent = truncateEntrypointContent(strippedContent).content
  }
  
  // 4. 检测是否与磁盘不同
  const contentDiffersFromDisk = finalContent !== rawContent
  
  return {
    info: {
      path: filePath,
      type,
      content: finalContent,
      globs: paths,
      contentDiffersFromDisk,
      rawContent: contentDiffersFromDisk ? rawContent : undefined,
    },
    includePaths,
  }
}
```

```typescript
 * 重新添加会话元数据到文件尾部

reAppendSessionMetadata(skipTitleRefresh = false): void {
  if (!this.sessionFile) return
  
  // 1. 读取尾部，吸收 SDK 可能写入的更新
  const tail = readFileTailSync(this.sessionFile)
  
  // 2. 从尾部提取可能更新的值（SDK 通过 API 修改）
  if (!skipTitleRefresh) {
    const tailTitle = extractCustomTitleFromTail(tail)
    if (tailTitle) {
      this.currentSessionTitle = tailTitle  // 吸收 SDK 更新
    }
  }
  
  // 3. 重新添加所有元数据
  void this.appendEntry({
    type: 'custom-title',
    title: this.currentSessionTitle,
    uuid: generateUuid(),
  })
  void this.appendEntry({
    type: 'tag',
    tag: this.currentSessionTag,
    uuid: generateUuid(),
  })
  void this.appendEntry({
    type: 'last-prompt',
    prompt: this.currentSessionLastPrompt,
    uuid: generateUuid(),
  })

```

**调用时机**：
- 每次 flush 后（清理程序）

## 7. Skill 内容恢复

### 7.1 Invoked Skills Attachment

```typescript
export function createSkillAttachmentIfNeeded(
  agentId?: string,
): AttachmentMessage | null {
  const invokedSkills = getInvokedSkillsForAgent(agentId)
  
  if (invokedSkills.size === 0) return null
  
  // 按调用时间排序（最近优先）
  let usedTokens = 0
  const skills = Array.from(invokedSkills.values())
    .sort((a, b) => b.invokedAt - a.invokedAt)
    .map(skill => ({
      name: skill.skillName,
      path: skill.skillPath,
      content: truncateToTokens(
        skill.content,
        POST_COMPACT_MAX_TOKENS_PER_SKILL,  // 5K per skill
      ),
    }))
    .filter(skill => {
      const tokens = roughTokenCountEstimation(skill.content)
      if (usedTokens + tokens > POST_COMPACT_SKILLS_TOKEN_BUDGET) {  // 25K total
        return false
      }
      usedTokens += tokens
      return true
    })
  
  if (skills.length === 0) return null
  
  return createAttachmentMessage({
    type: 'invoked_skills',
    skills,
  })
}
```

### 7.2 Per-Skill 截断

```typescript
const SKILL_TRUNCATION_MARKER =
  '\n\n[... skill content truncated for compaction; use Read on the skill path if you need the full text]'

function truncateToTokens(content: string, maxTokens: number): string {
  if (roughTokenCountEstimation(content) <= maxTokens) {
    return content
  }
  const charBudget = maxTokens * 4 - SKILL_TRUNCATION_MARKER.length
  return content.slice(0, charBudget) + SKILL_TRUNCATION_MARKER
}
```

```typescript
export async function createAsyncAgentAttachmentsIfNeeded(
  context: ToolUseContext,
): Promise<AttachmentMessage[]> {
  const appState = context.getAppState()
  const asyncAgents = Object.values(appState.tasks).filter(
    (task): task is LocalAgentTaskState => task.type === 'local_agent',
  )
  
  return asyncAgents.flatMap(agent => {
    if (
      agent.retrieved ||
      agent.status === 'pending' ||
      agent.agentId === context.agentId

      return []
    }
    return [
      createAttachmentMessage({
        type: 'task_status',
        taskId: agent.agentId,
        taskType: 'local_agent',
        description: agent.description,
        status: agent.status,
        deltaSummary:
          agent.status === 'running'
            ? (agent.progress?.summary ?? null)
            : (agent.error ?? null),
        outputFilePath: getTaskOutputPath(agent.agentId),
      }),
    ]
  })
}
```

**覆盖场景**：
- 正在运行的 agent（防止 spawn 重复）
- 已完成但未检索结果的 agent（保留结果）

## 9. Delta Attachments 重新注入

### 9.1 问题背景

压缩会丢弃之前的 delta attachments（tools、agents、MCP 指令）。压缩后需要重新注入这些 attachments，以便模型知道可用的工具。

## 10. 总结

文件状态缓存与恢复系统的关键设计：

| 机制 | 描述 |
|------|------|
| FileStateCache | 内存缓存最近读取的文件内容和时间戳 |
| 压缩后恢复 | 按时间排序恢复最多 5 个文件，受 50K token 预算限制 |
| 去重优化 | Preserved messages diff 避免重复恢复 |
| 排除规则 | 计划文件、CLAUDE.md 单独处理 |
| 内容差异化 | contentDiffersFromDisk 检测处理后的内容 |
| 元数据持久化 | JSONL 存储 custom title、tag 等 |
| 16KB 尾部窗口 | 快速读取元数据用于 --resume |
| Re-append | 压缩后重新添加元数据到尾部窗口 |
| 多类型恢复 | Files、Plans、Skills、Async Agents、Plan Mode |

这套系统确保了压缩和会话恢复后，模型仍然拥有必要的文件上下文和状态信息。
